## 계기

우아콘2025에서 배달의민족 팀의 "기록방" 발표를 봤다. QA 과정에서 버그를 신고할 때 화면 녹화 + 콘솔 로그 + 네트워크 로그를 묶어서 링크 하나로 공유한다는 아이디어였다. 발표를 보는 내내 "이거 우리 팀에도 딱 필요한 건데" 싶었다.

우리 팀은 신규 서비스 개발 중이었고, QA 단계에서 버그를 공유하는 방식이 늘 불편했다. 테스터가 스크린샷을 찍어 Jira에 올리면 개발자가 재현을 시도하는데, "저는 되는데요"가 반복되곤 했다. 스크린샷 한 장으로는 클릭 순서도, API 응답도, 콘솔 에러도 알 수 없으니까.

발표를 본 다음 날부터 비슷한 툴을 직접 만들어보기로 했다.

---

## 상용 툴은 왜 안 썼나

처음에는 Microsoft Clarity나 Hotjar 같은 서비스를 붙이는 게 가장 빠른 방법처럼 보였다. 그런데 몇 가지 이유로 포기했다.

**데이터 주권 문제.** 내부 업무 시스템이나 아직 출시 전인 서비스의 화면을 외부 서버로 전송하는 건 찜찜했다. 아직 공개되지 않은 기능이 녹화돼서 외부 서버로 나가는 상황은 피하고 싶었다.

**QA 워크플로우 커스터마이징 불가.** Clarity나 Hotjar는 마케팅/UX 분석 목적이지 QA 버그 리포팅 목적이 아니다. 심각도 분류, Jira 자동 연동, 완료 처리 같은 기능은 없다. 결국 원하는 기능을 다 만들려면 어차피 직접 연동해야 한다.

**비용.** 팀 내부 용도로 쓰기엔 월정액이 부담이다.

직접 만들면 세 가지를 모두 해결할 수 있었다. 오픈소스 라이브러리(rrweb)와 Supabase free tier 조합이라면 비용도 거의 없다.

## rrweb을 선택한 이유

화면 녹화 방식은 크게 세 가지를 생각했다.

**`MediaRecorder`로 비디오 녹화** — 가장 직관적이지만 파일이 크고, 재생 중 특정 시점으로 점프하기 어렵다. 콘솔 로그나 네트워크 로그와 ms 단위로 동기화하는 것도 불가능에 가깝다.

**Canvas 캡처** — `captureStream()`으로 캔버스를 녹화하는 방식인데, 결국 비디오라 동일한 문제가 있다.

**rrweb** — DOM 변경 사항을 이벤트 스트림으로 직렬화한다. 텍스트 기반이라 lz-string으로 압축하면 크기를 많이 줄일 수 있고, 각 이벤트에 타임스탬프가 붙어 있어서 콘솔/네트워크 로그와 정확히 동기화할 수 있다. `checkoutEveryNms`로 롤링 버퍼도 구현하기 쉽다.

rrweb의 압도적인 장점은 **"로그와의 동기화"**였다. 재생 중 어느 시점에 어떤 API가 호출되었고, 어떤 에러가 콘솔에 찍혔는지를 한 화면에서 같이 볼 수 있다. 비디오 녹화로는 이게 불가능하다.

## 전체 구조 한 줄 요약

```
타겟 앱 (qaroom SDK 주입)
  → 화면/콘솔/네트워크 캡처
  → 버튼 클릭 시 lz-string 압축 + Supabase 저장
  → 뷰어 URL 생성

QA 기록방 Viewer (React SPA)
  → Supabase에서 세션 로드
  → rrweb-player로 재생
  → 콘솔/네트워크 로그 동기화

Supabase Edge Function
  → Jira REST API 프록시 (CORS 우회 + 토큰 보안)
```

SDK는 타겟 앱에 `<script>` 태그 하나로 주입하거나 npm 패키지로 설치해 `initQaRoom()` 한 줄로 시작할 수 있다. 이후로는 완전히 백그라운드에서 동작한다.
