## 배경

2025년 10월, 우아한형제들이 주최하는 개발자 컨퍼런스 우아콘에 참석했다. 총 6개 세션을 들었고, 발표 주제는 3D 렌더링부터 디자인 시스템, 성능 최적화, QA 자동화까지 다양했다. 당시 업계 분위기는 아직 "AI는 강력한 도구"라는 수준이었다. 반년 후인 2026년 초와 비교하면, AI가 에이전트로 언급되기보다는 실무의 보조 수단으로 조심스럽게 자리잡는 중이었다.

세션을 들으면서 자연스럽게 한 가지 렌즈를 가지게 됐다. "이걸 우리 팀에 도입한다면 어떻게 될까?"

## 배운 내용

### QA 디버깅의 병목은 재현 과정에 있다

가장 인상 깊었던 세션 중 하나는 웹뷰 환경에서 QA 프로세스를 개선한 내부 도구 이야기였다.

기존 흐름은 익숙한 고통이었다. QA 담당자가 이슈를 발견하면, 재현하기 위해 처음으로 돌아가고, 찰스 같은 프록시 툴을 연결하고, 각종 정보를 수집해서 지라 티켓에 붙여넣는다. 개발자는 티켓을 받아 필요한 데이터가 없으면 다시 요청하고, QA는 재현을 시도하고... 이 왕복이 컨텍스트 스위칭을 만들고 두 직군 모두 지치게 한다.

이 팀이 만든 도구는 이 왕복을 없앴다. SDK가 웹뷰에서 실시간으로 데이터를 수집하고, 이슈를 발견하는 순간 버튼 하나로 "기록방"을 생성한다. 기록방에는 그 시점의 콘솔, 네트워크, 화면 캡처, 세션 리플레이, 사용자 정보가 자동으로 담겨 있다. 프론트엔드 개발자에게 친숙한 개발자 도구 형태로 렌더링되는 점도 영리했다.

구현의 핵심은 Chrome DevTools Protocol(CDP)이었다. 수집한 데이터를 CDP 형태로 매핑해두면, 구글의 오픈소스인 DevTools Frontend를 그대로 가져다 기록방 UI로 쓸 수 있다. 직접 개발자 도구를 만드는 게 아니라, 개발자 도구가 이해하는 언어로 데이터를 포맷하는 것이 핵심이었다.

이 접근이 좋았던 이유는, 문제를 "재현"의 문제가 아닌 "수집 자동화"의 문제로 재정의했다는 점이다. 이슈 등록 시간이 절반으로 줄었다는 QA 피드백보다, "앞으로 돌아가지 않아도 된다"는 단방향 흐름의 전환이 더 본질적인 변화였다.

### 디자인 시스템을 AST와 MCP로 연결하다

또 하나 바로 적용하고 싶었던 세션은 디자인 시스템 운영 이야기였다. 컴포넌트가 90개를 넘고, 서비스 개발자가 바쁜 일정에 쫓기다 보면 디자인 시스템에서 컴포넌트를 detach해서 인라인 스타일로 덮어쓰게 된다. 디자인 토큰 업데이트가 반영되지 않는 단절이 쌓이면, 결국 디자인 시스템이 있어도 없는 것과 다름없어진다.

이 팀이 택한 방향은 두 가지였다.

첫째는 AST 기반 분석과 코드모드 자동화다. Babel의 파서와 트래버스를 이용해 코드베이스에서 하드코딩된 색상값, 잘못된 컴포넌트 임포트를 찾아내고, 인터랙티브 CLI 도구로 디자인 토큰으로 자동 교체한다. 분석 결과는 대시보드로 시각화해 어느 서비스가 얼마나 이탈했는지 정량적으로 볼 수 있다.

둘째는 MCP 서버를 통한 맥락 제공이다. 흩어진 정보(공식 문서, 피그마 가이드, 스토리북 예제)를 마크다운으로 통합해 MCP 리소스로 만들고, 커서 같은 AI 에디터가 디자인 시스템 규칙을 직접 참조할 수 있도록 했다. 피그마 링크를 던지면 디자인 시스템 컴포넌트로 구현된 코드가 자동 생성되는 데모가 가장 인상적이었다.

문의 채널에 연간 1,300건이 넘는 티켓이 쌓이는 문제를, "자동화"가 아닌 "맥락의 재배치"로 풀어낸 점이 기억에 남는다.

### 성능 수치가 없는 최적화는 증명되지 않는다

장보기 서비스를 네이티브 앱에서 웹뷰로 전환한 세션은, 기술보다 반성을 더 많이 가져왔다.

이 팀은 SSR에서 CSR로 전환하며 FCP를 39% 줄였고, 이미지 지연 로딩과 리사이저 도입으로 네트워크 요청을 70% 감소, 로드 타임을 86% 단축했다. 캐시 미적용이라는 근본적인 설정 오류를 수정하자 AWS 비용이 월간 55% 절감되었고, 연간 3천만 원의 인프라 비용이 줄었다. 결과적으로 FCP·LCP를 1초 이내로 만들었다.

이야기를 들으면서 무릎을 쳤다. 나도 비슷한 최적화를 한 적이 있다. SSR 구조 개선, 이미지 최적화, 스켈레톤 도입. 그런데 수치가 없다. 빠르기는 빨라졌지만 "얼마나"를 말할 수 없다. 이 팀은 Sentry Performance로 실사용자 지표를 모니터링하고, 배포 후 성능이 특정 임계치 아래로 떨어지면 슬랙 알림을 받았다.

성능 최적화의 가치는 개선된 수치에 있다. 수치가 없으면 비즈니스 언어로 번역되지 않는다.

### 도전적이지만 아직은 먼 것들

3D 렌더링 세션은 Three.js로 샌드위치를 쌓는 리워드 프로그램 개발기였다. 2D 레이어와 3D 레이어를 분리하고, 단일 상태 기반으로 뷰 전환을 제어하는 설계가 흥미로웠다. 하지만 솔직히 "이걸 우리 서비스에서 쓸 일이 있을까?" 라는 생각이 먼저 들었다.

어드민 스튜디오 세션은 노코드 어드민 에디터를 만든 이야기였다. DSL로 어드민 구조를 정의하고, OpenAPI Spec을 통해 API를 표준화하며, IndexedDB로 임시 저장, 깃에서 영감을 받은 버전 관리까지 구현한 야심찬 프로젝트였다. 규모 있는 조직에서라면 강력한 솔루션이지만, 지금 내 팀의 어드민 볼륨에는 오버엔지니어링일 것 같았다.

로그 회귀 테스트 세션은 API 응답 변화를 스냅샷으로 저장하고 MSW와 결합해 회귀를 감지하는 이야기였다. "그렇군, 그런 문제가 쌓여서 이걸 만들었구나"라는 공감은 들었지만, 지금 당장 우선순위가 되지는 않았다.

## 느낀 점

이 컨퍼런스에서 AI는 조용한 배경이었다. 여러 세션에서 AI 도구를 활용했다고 언급했지만, 주인공은 어디까지나 문제와 해결책이었다. 반년 후 원티드 하이파이브 2026에서 AI가 모든 대화의 중심이 되었던 것과 비교하면, 업계의 무게중심이 얼마나 빠르게 이동했는지 실감하게 된다.

도입 가능성이라는 렌즈는 생각보다 좋은 필터였다. 인상적인 기술이 반드시 적용 가능한 기술은 아니었고, 우리 팀의 문제와 겹치는 지점에서 세션이 살아났다. 디버깅 도구와 디자인 시스템 MCP는 당장 참고할 수 있는 방향이었고, 성능 수치 부재에 대한 반성은 컨퍼런스 전체에서 가장 실용적인 수확이었다.

좋은 발표가 "이렇게 했더니 됐다"로 끝나는 데 반해, 오래 남는 발표는 "나는 왜 그러지 못했는가"라는 질문을 남긴다.
