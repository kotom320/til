## 배경 / 계기

운영툴 예약 도메인에 테스트 도입 타당성을 검토하다가, 이미 10개 넘는 테스트 파일이 존재한다는 걸 발견했다.
그런데 `pnpm test`를 돌려보니 **"No test files found"** — vitest config의 include 패턴(`*._test.tsx`)과 파일 네이밍(`*.test.tsx`)이 달라서 하나도 실행되지 않는 상태였다.

더 근본적인 문제는 파일이 실행됐다 해도 테스트 내용 자체가 신뢰를 주지 못한다는 점이었다.

## 핵심 개념 — 순환 논리 함정

기존 테스트들은 mock 데이터를 정하고, 그 mock에 맞는 기대값을 검증하는 구조다.

```ts
// mock: unitPrice 500, 1000으로 설정
const optionList = [
  { id: 1, unitPrice: 500 },
  { id: 2, unitPrice: 1000 },
]

// 옵션1 2개 + 옵션2 3개 선택
// 기대값: 4,000원 (500*2 + 1000*3)
expect(await screen.findByText('4,000원')).toBeInTheDocument()
```

이 테스트의 구조를 보면:

```
mock 데이터 설계 (개발자) → 코드 작성 (개발자) → 기대값 설정 (개발자)
```

세 가지를 같은 사람이 같은 시점에 만든다. 결국 "내 코드가 내 mock에서 내가 기대한 값을 뱉는다"를 확인하는 것에 불과하다. **자기 자신을 검증하는 순환 논리**다.

## 어디서 버그가 숨는가

이 구조에서 잡히지 않는 버그들이 있다.

**1. mock이 실제 API 응답과 다른 경우**

실제 API는 `unitPrice`를 문자열 `"500"`으로 내려주는데, mock은 숫자 `500`으로 설정했다면?
`Number("500") * 2`는 동작하지만, 코드가 타입 변환 없이 문자열 연산을 하면 `"500500"` 이 된다.
mock이 편의에 맞게 설계되어 있어서 이 버그가 테스트에 잡히지 않는다.

**2. 비즈니스 규칙이 코드에 잘못 구현된 경우**

차감 금액을 선택하면 총액이 감소해야 한다는 규칙이 있다.
개발자가 이걸 반대로 구현(양수로 더함)한 채로, mock과 기대값을 모두 그 잘못된 로직에 맞춰 작성하면? 테스트는 통과하고 버그는 살아있다.

**3. 엣지 케이스**

0원, undefined, 음수 입력 같은 경우는 편의 중심으로 만든 mock에 포함되지 않는다.

## 믿을 수 있는 테스트의 기준

> mock 데이터와 기대값이 서로 독립적으로 정당화될 수 있는가?

- **mock**: 실제 API 응답 스펙, 또는 실제 DB 데이터에서 도출
- **기대값**: 비즈니스 규칙(기획서, 정책 문서)에서 독립적으로 도출

둘이 서로 다른 출처에서 나와야 한다. 그래야 둘 중 하나가 틀렸을 때 테스트가 깨진다.

실용적으로는, **로직 변경 시 반드시 깨지는 테스트**가 믿을 수 있는 테스트다.

```ts
// 비즈니스 규칙을 명시한 테스트
test('차감 금액은 총액을 감소시킨다', () => {
  expect(calcArbitraryAmount('차감 금액', 3000)).toBe(-3000)
})

test('차감 금액을 선택하지 않으면 총액에 영향이 없다', () => {
  expect(calcArbitraryAmount(undefined, 3000)).toBe(0)
})
```

차감 로직이 반전되면 `-3000` 대신 `3000`이 나와서 즉시 실패한다.
"차감 금액을 선택하지 않은 케이스"는 편의 중심 mock에는 포함하지 않는 엣지 케이스지만 규칙 기반으로 접근하면 자연스럽게 나온다.

## 정리

테스트의 가치는 수가 아니라 "이 테스트가 통과하면 실제로 동작한다고 확신할 수 있는가"에 있다.
mock과 기대값을 같은 사람이 동시에 만들면 순환 논리가 되고, 구색은 맞지만 믿음은 생기지 않는다.
