## 1. 문제 인지 / 상황 설명

새로운 TF에서 다크모드 대응이 필요해졌고, 이를 계기로 디자인 시스템 구축이 시작되었다.

기존 방식은 다음과 같았다.

- 디자인 시스템을 별도 repository로 관리
- Rollup으로 번들링
- npm/pnpm 패키지로 배포 후 설치

하지만 이번 TF에서는 **빠른 개발 속도**가 가장 중요한 목표였다.
따라서 기존 방식 대신 다음 전략을 선택했다.

> 디자인 시스템을 별도 패키지로 분리하지 않고 현재 개발 중인 repo 내부에 직접 녹여 관리한다.

이를 통해 초기 개발 속도를 높일 수 있었다.

## 2. 문제 정의 및 원인 분석

### 2.1 디자인 시스템 구성 기준

디자인 시스템은 다음 기준으로 구성했다.

- 컬러와 Typography는 **Design Token**으로 관리
- 나머지 UI 요소는 컴포넌트화하여 재사용성 확보

이 구조를 통해 다크모드 대응 시 토큰 변경만으로 전체 UI를 일관되게 조정할 수 있게 했다.

### 2.2 초기 의사결정: Storybook 제외

빠른 개발을 위해 초기에는 Storybook을 개발 범위에서 제외했다.

- 설정 비용 존재
- 초기 컴포넌트 수가 적음
- TF 일정이 촉박함

즉 “속도 > 문서화”라는 판단이었다.

### 2.3 문제: 결과 공유의 어려움

개발이 진행될수록 새로운 문제가 나타났다. 디자이너에게 구현 결과를 공유할 방법이 부족했다.

기존 방식:

- 로컬 화면 캡처 공유
- 개발 서버 화면 공유
- 특정 페이지에서만 확인 가능

문제점:

- 컴포넌트 단위 확인 불가
- 상태별 UI 확인 어려움
- 반복적인 커뮤니케이션 발생

결국 다음 질문에 도달했다.

> 디자이너가 직접 확인할 수 있는 UI 환경이 필요하지 않을까?

## 3. 해결 방법

### 3.1 해결 방법 탐색

여러 방법을 고민했다.

- HTML을 정적으로 빌드하여 제공
- 다른 UI 문서화 도구 도입
- 개발 환경 전용 route 페이지 생성
- Storybook 재도입

검토 결과 Storybook이 가장 합리적인 선택이라고 판단했다.

- 컴포넌트 단위 확인 가능
- 상태별 UI 표현 가능
- 디자인 시스템 구조와 자연스럽게 연결
- 장기 유지보수에 유리

### 3.2 Storybook + Chromatic 도입

Storybook 도입을 팀원들과 공유하고, Chromatic을 통해 배포하는 방식에 대해 협의 후 진행했다.
repo가 회사 소유의 private 환경이었기 때문에 Chromatic 접근 권한 설정도 함께 진행했다.

### 3.3 Chromatic 배포로 달라진 점

Chromatic을 통해 Storybook을 배포함으로써 다음이 가능해졌다.

- 디자이너가 별도 환경 없이 UI 확인
- 배포된 컴포넌트 기준 공유
- 링크 기반 피드백 가능

이로써 UI가 코드 내부가 아닌 **공유 가능한 산출물**이 되었다.

## 4. 결과

디자이너가 직접 확인할 수 있는 Storybook 환경을 제공했고 협업 경험은 전반적으로 만족스러웠다.

특히 다음 변화가 있었다.

- 구현 결과 공유 과정 단순화
- UI 확인 커뮤니케이션 감소
- 디자인 검증 접근성 향상

## 5. 결론 (배운 점)

- 디자인 시스템은 기술보다 협업 문제에서 시작된다.
- 빠른 개발을 위해 생략한 도구라도, 협업 단계에서 다시 필요해질 수 있다.
- Storybook은 문서화 도구라기보다 협업 인터페이스에 가깝다.
- Chromatic은 배포 도구라기보다 UI 커뮤니케이션 채널 역할을 한다.

결과적으로 이번 과정은,

> 디자인 시스템은 컴포넌트를 만드는 일이 아니라 협업 방식을 구축하는 일이라는 것을 이해하게 된 과정이었다.

## 6. 다음 스텝

현재는 Storybook 배포까지 완료된 상태이며, 향후 Chromatic의 Visual Review 기능을 활용해 다음을 구축할 예정이다.

- PR 단위 UI 리뷰
- 시각적 변경 검증
- 개발자와 디자이너 간 UI 리뷰 프로세스
