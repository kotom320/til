## 문제 상황

CloudFront 이미지 URL에서 특정 파일만 간헐적으로 조회되지 않는 현상이 발생했다.

이 프로젝트의 CloudFront는 base64 인코딩된 JSON 토큰을 path로 사용해 S3 오브젝트를 서빙한다.

```
https://your-cloudfront-domain.cloudfront.net/{base64_token}
```

토큰을 디코딩하면 아래와 같다.

```json
{"bucket": "your-bucket-name", "key": "public/icons/some-icon.png"}
```

같은 버킷, 같은 경로 구조인데 `icon-a.png`는 정상 조회, `icon-b.png`는 403이 났다.

## 원인 분석

### 1단계: 토큰 구조 비교

두 URL의 base64 토큰을 디코딩해 비교했다. 구조가 완전히 동일했다. S3 ACL 차이, CloudFront 캐시 히트 여부 등을 의심했지만 모두 추정에 불과했다.

### 2단계: 패딩 여부에 따른 테스트

토큰 뒤에 `=`를 붙여가며 테스트했다.

| 토큰 끝 | 결과 |
|--------|------|
| `...fQ` (패딩 없음) | ❌ 403 |
| `...fQ=` (1개) | ✅ 200 |
| `...fQ==` (2개) | ✅ 200 |

패딩 유무만으로 결과가 달라졌다.

### 3단계: 토큰 길이 확인

```
icon-b 토큰 길이: 122  →  122 % 4 = 2  →  == 2개 필요
icon-a 토큰 길이: 123  →  123 % 4 = 3  →  = 1개 필요
```

`icon-a`는 패딩이 1개 필요한 길이라 누락돼도 Lambda@Edge 디코더가 허용했다. `icon-b`는 2개가 필요한데 없어서 디코딩 실패 → 403.

## 원인

백엔드에서 이미지 토큰을 DB에 저장할 때 base64 패딩(`=`)을 제거하고 저장하고 있었다. 클라이언트는 저장된 토큰을 그대로 URL에 붙여 사용하므로, 패딩이 필요한 길이의 토큰에서만 403이 발생했다.

JavaScript의 `btoa()`는 항상 올바른 패딩을 붙여주므로 프론트 생성 URL에서는 문제가 없었다. DB에 저장된 백엔드 생성 토큰에서만 누락됐다.

## 해결 방법

백엔드 토큰 저장 시 패딩을 포함하도록 수정했다.

패딩이 필요한 자릿수는 아래 공식으로 계산한다.

```
필요한 = 개수 = (4 - (길이 % 4)) % 4
```

프론트에서 임시 대응이 필요하다면 토큰 사용 전 정규화할 수 있다.

```typescript
const pad = (token: string) => token + '='.repeat((4 - (token.length % 4)) % 4);
```

## 정리

- base64 패딩 누락은 "가끔 되고 가끔 안 되는" 증상을 만든다. 디코더 구현마다 관용도가 달라서 1개 누락은 통과해도 2개 누락은 거부할 수 있다.
- 비슷한 URL인데 일부만 실패할 때는 토큰 길이와 패딩 여부를 먼저 확인하는 게 빠르다.
- 근거 없는 추정(ACL 차이, 캐시 여부)보다 실제로 측정 가능한 것(토큰 길이, `% 4` 나머지)을 먼저 보자.
